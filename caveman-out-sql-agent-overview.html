SQL-Agent

Dynamic, schema-agnostic natural-language → SQL with execution-driven self-correction · high-level overview · 2026-06-20

What it is — in one breath

Drop in any SQLite database. The system compiles its schema into a compact in-prompt cache, a fine-tuned small model drafts SQL, runs it, and fixes its own query from the real error — up to 3 tries. No schema is memorized in the weights. It is the generalization of the sibling nanoSQL project, which memorized a single schema.

How we build the training data

Target: ~15k verified examples · 60% clean / 40% self-correction · spanning ~210 real databases.

flowchart TD S[Spider 744 DBs + gold SQL] --> G[Gold pairs] B[BIRD dev 11 DBs + gold SQL] --> G G --> CL[Clean set 60 percent
reuse gold, no LLM generation] CL --> V1{Execute vs gold
result-set match} G --> GEN[Correction set 40 percent
haiku + sonnet waves] GEN --> WRONG[Capture a wrong attempt
+ real traceback] WRONG --> V2{Final fix matches gold} V1 -->|pass| POOL[Verified pool] V2 -->|pass| POOL POOL --> DIV{Dense + diverse?
per-DB + query-type + difficulty} DIV -->|no, backfill gaps| GEN DIV -->|yes, 15k reached| OUT[train / val / test JSONL]
  • Clean 60% comes from the human-verified gold SQL the datasets already ship — reformatted, not generated. Cheaper, higher quality.
  • Correction 40% is the real generation work: haiku does cheap high-volume candidates, sonnet handles hard fixes. Every example is execution-verified against the real database.
  • Validity = result-set matches gold (not merely "non-empty"). Tracebacks are real, never invented.
  • Stored as model-agnostic message lists, rendered to each model's chat template at train time.
